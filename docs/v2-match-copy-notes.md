# V2 匹配卡片文案补充

## 本轮调整

- 将 V2 匹配亮点改成更适合标签芯片展示的短句，减少“整句判断”感。
- 将 V2 破冰建议保留在可自然接话的范围内，避免过度像心理测试追问。
- 当前台同时拿到模板文案、派生文案和底层算法文案时，优先展示模板与派生文案，再回退到底层算法产物。

## 目标

- 让匹配页更像“关系风格提示”，而不是“算法宣判”。
- 让用户看到亮点时能快速抓住感觉，看到破冰建议时能直接拿去开口。

## 代码位置

- `E:\code\datematch-main\app\data\matchContent.ts`
- `E:\code\datematch-main\lib\match-helpers.ts`

## 动作区文案

- 匹配卡片顶部与确认区改为按模式切换文案。
- 恋爱模式更偏“靠近 / 点亮 / 继续了解”。
- 友情模式更偏“回应 / 继续认识 / 接住连接”。
- 这样可以让结果页、匹配亮点和互动动作保持同一套关系语言。

## 分数解释区

- 匹配卡片中的 breakdown 区域已改成用户可读语言，不再直接展示“性格匹配 / 背景相似 / 互补程度”这类工程化字段名。
- 恋爱模式使用“关系理解 / 相处入口 / 现实落点 / 节奏互补”。
- 友情模式使用“朋友节奏 / 共同话题 / 生活场景 / 相处余地”。
- 每个分项都补了一句解释，让用户知道这个分数在关系里实际代表什么。

## 词表单源

- 匹配算法中的维度中文名已改为复用 `app/data/matchContent.ts` 中的 trait label 表。
- 这样结果页、匹配卡片、算法兜底亮点与建议可以共用同一套维度命名，减少后续改词时的分叉。
- 当前已覆盖恋爱与友情两套 `legacy / v2` 标签映射。

## 画像语义映射

- 在 V2 总画像配置中新增了 `signatureTags`，作为结果页画像与匹配页亮点之间的中间语义层。
- 匹配页在生成亮点时，会优先尝试读取双方画像的共同语义标签，而不只是依赖单个 trait 命中。
- 第一版先覆盖“共同画像气质接近”和“共享语义标签”两种亮点来源，先把结果页与匹配页连起来。

## 母词表

- V2 画像的 `signatureTags` 已从自由短词改成受控母词引用。
- 当前在 `resultsContent.ts` 中拆成恋爱与友情两套母词表，后续继续改词时应优先改母词，不再直接散改画像数组里的短语。
- 这一层的目标是让结果页画像、匹配亮点和建议文案都从同一组关系语义出发。

## 映射规范

- 每个 V2 画像现在除了 `signatureTags`，还带有 `themeProfile`：
  - `core`：最能代表这个画像的核心语义。
  - `secondary`：可辅助出现的次级语义。
  - `avoid`：不应混入这个画像气质的词域。
- 匹配页在生成画像语义亮点时，会优先取双方共同的 `core`，再尝试共同 `secondary`，并避开双方 `avoid` 中出现的词。
- 这样做的目标是减少画像之间的“串味”，让匹配页更像结果页画像的延伸，而不是另一套随机标签系统。

## 主题词渲染层

- 目前又加了一层 theme token -> match chip copy 的渲染映射，位置在 lib/match-helpers.ts。
- 这一层的职责不是重新定义画像，而是把结果页内部使用的语义母词，翻译成匹配卡片里更自然、更像成品的短句。
- 例如不再直接把 稳定回应、边界清楚 这类母词裸露给用户，而是转成：
  - 都重视稳定回应
  - 亲近里也重视边界
  - 都偏好低压陪伴
- 这样结果页仍然保留统一语义底座，匹配页则获得更像“关系提示”的可读表达。

## 高亮去重规则

- 由于匹配卡片同时会混合：
  - 画像语义亮点
  - 模板亮点
  - 派生亮点
  - 算法兜底亮点
- 当前在 getHighlights 中新增了一层语义近似去重。
- 去重不是只看完全相同的字符串，也会尽量合并像“画像比较接近 / 关系理解比较接近”这类过于接近的提示，避免一张卡片里重复表达同一种意思。
- 这一步是轻量规则，不追求 NLP 级别准确，目标只是把用户第一眼看到的四个 chip 控制得更干净。


## 画像级句式

- 现在“双方命中同一 V2 画像”时，不再统一显示“画像比较接近 / 气质比较接近”。
- 已按画像 id 增加一层 profile-level match copy，让不同画像在匹配页上更有区分度。
- 例如：
  - stable_connector 会更偏“都想把关系落在稳定回应里”
  - present_experience 会更偏“都更在意相处里的真实感觉”
  - deep_companion 会更偏“都更想要长期理解而不只是热闹”
- 这样做的目标，是让匹配页在用户第一眼感受上，就能延续结果页总画像的个性，而不是只显示抽象的“接近”。

## 互补句式

- 现在 V2 的互补型高亮不再只统一显示“靠近方式有互补 / 相处节奏有互补”。
- 已在 lib/match-helpers.ts 中按互补维度增加一层 complementary highlight copy。
- 例如：
  - 恋爱里的 approachPace 会更偏“一个更主动，一个更稳，靠近节奏有互补”
  - 恋爱里的 reassuranceNeed 会更偏“一个更需要确认，一个更能给稳定回应”
  - 友情里的 connectionFrequency 会更偏“一个更常联系，一个更低压，友情频率有互补”
- 这一步的目标，是让“你们不一样”被用户读成可理解的互补关系，而不是抽象的算法结论。

## V2 破冰分层

- 现在 V2 的破冰建议不再只依赖通用模板，也会优先生成三类语义 opener：
  - 同画像 opener
  - 同主题 opener
  - 互补型 opener
- 生成位置在 lib/match-helpers.ts 的 generateIceBreakers 链路中，当前顺序是：
  - V2 语义 opener
  - 共同兴趣 opener
  - 偏好标签 opener
  - 模板 opener
- 这样做的目的，是让用户先看到和当前这张匹配卡“为什么成立”直接相关的开场方式，再看到泛化模板。
- 现在匹配页的高亮和破冰，已经共用同一套画像、主题词、互补维度语义，不再是两套独立语言系统。

## 互补选择逻辑

- 目前互补型展示不再简单取 complementaryTraits 的原始首项，而是在 buildMatchContentContext 阶段先做展示排序。
- 当前排序会同时参考：
  - 维度的展示优先级
  - 双方在该维度上的差值幅度
- 这样做的目标，是优先选出“最值得对用户说”的互补点，而不是机械地选出数组里的第一个差异维度。
- 这层排序会同时影响：
  - 匹配高亮里的互补句式
  - 互补型 opener
  - 模板里的 traitPair 占位符
- 第一版暂时仍然是轻量规则，但已经比原先的首项命中更稳定，也更贴近产品表达需要。

## 高亮组合策略

- 当前 getHighlights 不再只是把多路来源平铺后截断，而是先按类别做一次组合。
- 目前的高亮类别分为：
  - profile：画像层命中
  - complementary：互补层命中
  - shared：共同关系偏好
  - context：现实入口与共同背景
  - fallback：模板和旧算法兜底
- 当前策略会优先尽量保证一张卡里覆盖：
  - 一个“你们为什么像”
  - 一个“你们哪里互补”
  - 一个“你们共同看重什么”
  - 一个“你们现实里怎么接得上”
- 如果某一类没有合适候选，再从 fallback 中补齐。
- 这样做的目标，是让 4 个 chip 更像编辑过的组合，而不是多路命中后重复表达同一件事。

## 轻量 Trace 对齐

- 当前高亮选择已经开始参考 breakdown 支撑，而不只是文案命中本身。
- 目前会先把高亮粗略映射到四类依据：
  - personality
  - interests
  - background
  - complementary
- 然后按对应 breakdown 分值做一次轻量排序与过滤。
- 例如：
  - 互补句式会优先要求 complementary 分项足够支撑
  - 共同兴趣类文案会更依赖 interests 分项
  - 同画像 / 同主题类文案会更依赖 personality 分项
- 这一步还不是完整的显式 trace 数据结构，但已经先把“页面上说什么”和“底层哪里在支撑它”拉得更近。

## Trace 能力保留

- Highlight trace 的底层能力仍保留在 match-helpers.ts 中，可继续用于后续调试和迁移。
- 由于 dev-channel-2 现在已经是正式用户页面，trace 展示 UI 已从该页面移除，避免调试信息进入正式体验。
- 后续如果需要可视化查看高亮 trace，建议迁移到 ops 控制台中展示。


## Favicon 约定

- 当前站点 favicon 统一使用 public/favicon.ico。
- 不再在 app/ 目录下保留同名 favicon 文件，避免和 App Router 的 metadata file route 冲突。
- 这次实际问题表现为：
  - /favicon.ico 返回 500
  - 连带出现前端资源请求异常，看起来像整页崩溃
- 如果后续需要更新站点图标，优先替换 public/favicon.ico，并保持 layout metadata 中的图标地址继续指向 /favicon.ico。

## 结果页末尾关系卡片

- 结果页末尾“你更容易和什么样的人走近 / 走得久”这张卡片，现已补齐 V2 适配。
- 之前这块仍在读取 legacy 的 modeConfig.bestMatches / challengingMatches，因此会出现总画像是 V2，但最后一张卡片还是旧话术的问题。
- 现在改为按 resolveV2ProfileCopy 解析出的画像 id，从 V2 compatibility 映射中读取：
  - bestMatches
  - challengingMatches
- 这样恋爱模式与友情模式的末尾卡片，都能和当前 V2 总画像保持同一套关系语言。

## 结果页末尾卡片语气收口

- 在 V2 适配的基础上，结果页末尾这张关系卡片又补了一轮语气收口。
- 当前不再只展示两组列表标题，而是增加了：
  - 一句画像延伸式的 section intro
  - best / challenging 两侧各自的前导语
- 同时标题也从偏功能性建议口吻，调整为更接近关系画像的表达。
- 这样用户在阅读这张卡时，会更像在看“这个画像会被什么样的关系吸引 / 会在哪类关系里卡住”，而不是在看一份泛化建议清单。

## Next 产物目录隔离

- 当前已将 Next 的开发态和构建态产物目录分开：
  - next dev 使用 .next-dev
  - next build 使用 .next
- 这样做是因为此前在同一个工作区里反复执行 build 和 dev 时，.next 内的服务端 chunk 布局会互相污染，导致开发态出现：
  - Cannot find module './9161.js'
  - /favicon.ico 500
  - /_next/static/css/app/layout.css 404
- 现在 next.config.mjs 已按 phase 区分 distDir，后续开发时不需要再让 dev 和 build 共用同一份 .next。

## Ops Trace 迁移

- Highlight trace 的可视化查看现在已经迁移到 `ops` 页面，不再出现在 `dev-channel-2`。
- `ops` 会按当前轮次真实 pair 抽取少量样本，并从“某个用户看到某个匹配对象”的视角展示：
  - highlights
  - ice breakers
  - trace bucket / evidence
  - 当前画像在结果页末尾卡片中的 best / challenging 文案
- 这样做的目的，是让运营和内容调整都基于真实落地样本，而不是只盯着静态配置改词。

## 本轮真实样本巡检

- 以当前轮次真实数据巡检时，恋爱模式暂时没有可用 pair，友情模式存在 1 组真实 mutual pair，可展开成 2 个用户视角样本。
- 当前友情样本的 breakdown 与 trace 基本一致：
  - 互补类高亮由 complementary 支撑
  - 同校与日常接近类高亮由 background 支撑
  - 共性类高亮主要由 personality 支撑
- 从真实样本里看，问题主要不在“算法说错了”，而在少数句子仍然偏模板口吻：
  - 个别 opener 还像在继续做题，而不是自然开场
  - 结果页末尾卡片中少数“更适合你 / 更能走久”句式略显结论先行

## 巡检后微调

- opener 侧，本轮收口了两句真实样本中命中的高频模板：
  - 将“如果要选一种最舒服的相处感...”改得更像真实聊天入口
  - 将“一个偏主动一个偏接球...”改成更通用、不带运动隐喻的表达
- 结果页末尾卡片侧，本轮只微调了个别句子：
  - friendship / balanced_realist
  - romance / clear_autonomy
  - romance / gentle_repair
  - romance / present_experience
- 调整原则是：
  - 少用“适合你 / 更能走久”这类直接判断
  - 多写成“更容易让你放心靠近 / 更容易把关系放长 / 更容易让你把心放下来”这类关系体验语言
